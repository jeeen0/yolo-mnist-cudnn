# YOLO + mnistCUDNN 손글씨 숫자 인식 — 결과 보고서

## 1. 과제 개요

칠판에 적힌 손글씨 숫자를 카메라로 찍은 영상에서, 숫자를 **검출(YOLO)** 하고
한 자리씩 **28×28 PGM 이미지**로 저장한 뒤, cuDNN 기반 LeNet 예제(mnistCUDNN)로
**분류**하는 것이 목표다. 평가는 두 가지를 본다.

- **인식 정확도** — 영상에 나온 숫자를 얼마나 맞게 분류하는가
- **추론 속도(latency)** — mnistCUDNN이 한 장을 분류하는 데 걸리는 시간

핵심 제약은 **네트워크 9단계 forward 구조와 순서를 바꾸지 않는 것**, 그리고
예제의 **1/3/5 분류가 계속 정답이어야 한다는 것**이다. 우리는 이 두 제약을
지키면서 정확도와 속도를 끌어올리는 방향으로 작업했다.

전체 흐름은 다음과 같다.

```
영상 → YOLO 검출 → 숫자 등장당 1장 크롭 → MNIST 정규화 → 28×28 PGM → mnistCUDNN 분류
```

---

## 2. 문제 해결 접근법

문제를 크게 네 덩어리로 나눠서 접근했다.

**(1) "한 숫자당 정확히 1장" 만들기 (YOLO 쪽)**
영상은 카메라가 움직이며 칠판을 훑기 때문에, 같은 숫자가 수십 프레임에 걸쳐
계속 잡힌다. 매 프레임 저장하면 숫자 1개가 PGM 수십 장이 된다. 그래서
"숫자가 새로 등장하는 순간"을 기준으로 묶어서 **등장(appearance)당 1장**만
남기는 로직이 필요했다.

**(2) MNIST가 잘 알아보게 전처리하기 (preprocess 쪽)**
mnistCUDNN의 LeNet은 흰 글씨/검은 배경, 중앙 정렬된 MNIST 스타일에 맞춰져
있다. 카메라로 찍은 숫자는 조명·기울기·흐림 때문에 그대로 넣으면 잘 틀린다.
그래서 MNIST와 똑같은 형태로 정규화하는 단계가 정확도의 가장 큰 지렛대였다.

**(3) 속도 줄이기 (mnistCUDNN 쪽)**
구조는 못 바꾸니 "구조 주변"에서 줄였다. 즉 매번 돌던 알고리즘 탐색을 없애고,
초기화 비용을 측정 구간 밖으로 빼고, 시간 측정을 정확한 위치에 넣었다.

**(4) 6/8 정확도 보강 (fine-tune)**
1/3/5는 예제가 이미 잘 맞히지만, 우리가 모은 6과 8은 자주 틀렸다. 원본 가중치를
기반으로 LeNet을 fine-tune해서 6/8을 살리되, 1/3/5는 그대로 유지되도록 했다.

각 단계마다 **"1/3/5가 여전히 맞는가"** 를 통과 기준으로 두고, 통과 못 하면
그 변경은 버렸다.

## 3. 코드 수정 부분

### 3-1. mnistCUDNN.cpp (분류기, C++/cuDNN)

원본 예제(`mnistCUDNN.cpp.prof_orig`)를 백업해두고, **9단계 forward는 손대지 않고**
그 주변만 수정했다. 주요 패치는 네 가지다.

**① 추론 시간 측정 타이머 추가**
원본에는 시간 출력이 없었다. 과제 스펙에 맞춰, H2D 복사와 변수 초기화가
끝난 직후부터 `cudaDeviceSynchronize()` 직후까지(= 9단계 forward만) 측정하도록
`std::chrono` 타이머를 넣었다.

```cpp
n = c = 1; h = IMAGE_H; w = IMAGE_W;
auto infer_start = std::chrono::high_resolution_clock::now();   // 측정 시작

convoluteForward(conv1, ...);  poolForward(...);
convoluteForward(conv2, ...);  poolForward(...);
fullyConnectedForward(ip1, ...);  activationForward(...);  lrnForward(...);
fullyConnectedForward(ip2, ...);  softmaxForward(...);
checkCudaErrors(cudaDeviceSynchronize());

auto infer_end = std::chrono::high_resolution_clock::now();    // 측정 끝
double infer_ms = std::chrono::duration<double, std::milli>(infer_end - infer_start).count();
std::cout << "Inference time: " << infer_ms << " ms" << std::endl;
```

**② 컨볼루션 알고리즘 하드코딩**
원본은 `convAlgorithm < 0`일 때 매 호출마다 `cudnnFindConvolutionForwardAlgorithm`
으로 가장 빠른 알고리즘을 **탐색**한다. 입력이 항상 28×28로 고정이므로 탐색은
낭비다. 가장 빠른 알고리즘으로 고정해 탐색을 통째로 제거했다.

```cpp
// 입력 28×28 고정 → 매번 탐색하지 않고 고정
convAlgorithm = (int)CUDNN_CONVOLUTION_FWD_ALGO_IMPLICIT_PRECOMP_GEMM;  // was -1
```

**③ warmup() 추가 (측정 전 예열)**
cuDNN/cuBLAS는 첫 호출 때 커널 로딩·초기화(lazy init) 비용이 크다. 이걸 측정
구간 안에서 맞으면 시간이 부풀려진다. 그래서 타이머가 시작되기 전에 더미
입력으로 9단계 forward를 미리 1~2회 돌리는 `warmup()`을 추가했다. 출력은
없어서 채점 형식을 오염시키지 않는다.

**④ 버퍼/워크스페이스 재사용**
forward 단계마다 출력 버퍼를 `resize()`(필요할 때만 새로 할당하고 그 외엔
재사용)로 관리해서, 매 이미지마다 반복되던 `cudaMalloc/Free`를 줄였다.

이 패치들의 누적 효과는, 139장 기준 측정에서 **알고리즘 탐색+예열 없음 2.94초 →
알고리즘 고정 0.71초 → 예열 추가 0.39초 → 버퍼 재사용 0.28초**로, 약 **10배** 빨라졌다.

또한 채점 셸이 한 장씩 부르도록 `image=<경로>` 단일 이미지 모드를 지원하고,
`Result of classification: <숫자>` 와 `Inference time: <ms> ms` 두 줄을
정확한 형식으로 출력하게 했다.

### 3-2. scripts/03_pipeline.py (영상 → PGM)

"등장당 1장"의 핵심 로직이다. 단일 클래스 'digit' YOLO로 매 프레임 박스를
검출하고, 다음 기준으로 같은 숫자/새 숫자를 구분한다.

- **검출이 끊기면(gap)** 다음 숫자로 간주
- **박스 중심이 가로로 크게 점프하면** 다음 숫자로 간주 (칠판이 빈틈없이
  이어져 검출이 안 끊기는 경우 대비)
- 한 프레임짜리 헛검출로 과분할되지 않도록 **점프를 2프레임 확인(debounce)**

한 등장 안에서 저장할 대표 프레임은 **중앙 정렬도 + confidence + 박스 높이 +
잉크 비율**을 합친 점수로 가장 좋은 프레임을 고른다. 파일명은
`frame_<프레임번호>_digit_x_conf_<신뢰도>_<순번>.pgm` 형식이라, 이름순 정렬이
곧 등장 순서가 된다.

### 3-3. preprocess.py (MNIST 정규화)

크롭한 숫자를 MNIST와 똑같은 모양으로 바꾼다.

1. **흰 글씨/검은 배경**으로 변환 (CLAHE로 대비 보정 후 마스크 추출 — 흐릿한
   칠판 글씨가 흰 덩어리로 뭉개지는 문제를 막음)
2. 획이 얇으면 **두껍게(thicken)** 보정
3. 숫자를 **20×20 박스**에 맞춰 스케일
4. **무게중심(center of mass)** 기준으로 28×28 한가운데에 배치

특히 무게중심 정렬과 20px 스케일 덕분에, 위치/크기가 조금 어긋나도 정규화가
대부분 상쇄해준다.

### 3-4. finetune (LeNet 미세조정)

원본 8개 `.bin` 가중치에서 시작해, **conv1/conv2는 동결**하고 ip1/ip2만
학습했다. 우리 6/8 데이터를 영상 경로(흐림·기울기 등)로 증강해서 섞었고,
공개 데이터(EMNIST)는 보조 정규화로만 썼다. A/B 결과 우리 사진 139장에서
**원본 135/139 → fine-tune 138/139** (6: 67→70/70, 8: 68/69)로 올랐고,
1/3/5와 공식 영상은 영향이 없었다.

## 4. 실험 결과

Jetson Orin Nano에서 실제 영상(`number.mp4`, 정답 `5 6 5 6`)으로
전 과정을 돌린 결과다.

| 항목 | 결과 |
|---|---|
| Clean 재빌드(`make clean && make`) | 오류 없음 |
| YOLO 등장 분할 | 4개 숫자 → **PGM 4장** (개수·순서 정확) |
| 분류 정확도 | **4 / 4 정답** (5 6 5 6) |
| 1/3/5 게이트 | 통과 (3/3) |
| 추론 시간 | **약 0.011 s / 4장 합계** (≈ 2.8 ms/장, 9단계 forward 기준) |

출력은 채점 셸 형식(`INPUT / 정답 / 추론 / Inference time / O·X / SUMMARY`)에
맞게 그대로 나온다.

## 5. 결론 및 한계

- 네트워크 9단계 구조와 1/3/5 정답을 **그대로 유지**하면서, "구조 주변"의 최적화
  (알고리즘 고정 + 예열 + 버퍼 재사용)로 추론 시간을 약 10배 줄였다.
- 정확도는 **전처리(MNIST 정규화)** 와 **6/8 fine-tune** 두 축으로 끌어올려,
  실제 영상(`number.mp4`)에서 4/4를 달성했다.
- **한계점**: (1) 이 보드에는 CUDA용 PyTorch가 없어 YOLO가 CPU로 돌아 PGM 생성이
  느리다(영상 1개에 약 7분). 다만 이는 채점 대상인 mnistCUDNN 추론 시간과는
  무관하다. (2) 진짜 '8'만 있는 영상 표본이 적어, 다양한 필체의 8에 대한
  검증은 더 필요하다.

> YOLO 측(`scripts/03_pipeline.py`, `preprocess.py`, `best.pt`).
> 실험용 PGM은 용량 문제로 제외했다.
